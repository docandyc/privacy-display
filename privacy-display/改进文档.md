> 基于对当前 PoC 代码的逐文件审查，本文档列出**真实存在**（而非泛泛建议）的改进点。
> 每条改进包含：**现状/问题 → 改动位置 → 具体方案 → 验证方法 → 优先级/工作量**。
> 优先级依据用户重点（交底书特性补齐）排序：P0 > P1 > P2。

## 1. P0 — 交底书特性补齐（用户重点）

### A1. HDR 亮度补偿（ICtCp 软裁剪 + PQ/HLG）

**现状/问题**
交底书 4.3 节详细描述了 HDR 模式非线性补偿（PQ/ST.2084 空间、`L_k_comp=min(L_max, L_k·n)`、ICtCp 软裁剪保色相）。但当前 `subframe_composer.py` 只实现了 SDR 背光提升模型（γ=1），完全没有 HDR 路径。亮度提升导致的超色域问题（交底书提到用 ICtCp 软裁剪解决）也未处理。

**改动位置**
- 新建 `src/core/hdr_compensation.py`
- 修改 `src/core/subframe_composer.py`：`SubframeComposer` 增 `hdr_mode` 参数

**具体方案**
1. PQ (ST.2084) EOTF/OETF 编解码：`pq_encode(L_nits) → [0,1]`、`pq_decode`
2. RGB↔ICtCp 转换（经 LMS 空间）
3. 超色域软裁剪：在 ICtCp 空间保持 I（亮度）与色相角不变，仅压缩超界饱和度
4. HDR 子帧补偿：线性光空间中 `L_k·n`，峰值钳到 `L_max`（1000–4000 nits）

**验证方法**
- PQ 往返误差 `|pq_decode(pq_encode(L))−L| < 1e-3`
- 软裁剪后色相分量 ΔH≈0（仅饱和度变化）
- n=8 时 HDR 补偿后峰值仍在物理范围

**优先级/工作量**：P0 / 中（纯数学，约 200 行）

---

### A2. 黑帧防御 + 自动曝光攻击模型

**现状/问题**
交底书 3.2.4 提"每第 n+1 个时隙插入全黑帧，使相机 AE 误判环境亮度→后续帧过曝"；5.4 步骤17 提"帧生成延迟时插黑帧应急保护"。当前 `subframe_composer.py` 只有反色帧（`compose_inversion_frame`），无黑帧；`timing_controller.py` 无应急黑帧逻辑；`camera_simulator.py` 无 AE 攻击建模，因此"黑帧使相机过曝"这一防御**完全没有被验证**。

**改动位置**
- `src/core/subframe_composer.py`：增 `compose_black_frame()`
- `src/core/timing_controller.py`：增"渲染超时→应急黑帧"状态
- `src/attack/camera_simulator.py`：增 `auto_exposure_attack()`，建模相机 AE 增益自适应

**具体方案**
1. 黑帧 = 全 0；在周期末按策略插入，时长 `α·Δt`
2. AE 模型：相机依据近 N 帧平均亮度调整增益 `gain = target_lum / observed_lum`，黑帧拉低 observed_lum → gain 上升 → 后续真实子帧过曝饱和
3. 对比黑帧/无黑帧两种情况下 AE 攻击后的 OCR

**验证方法**
- 插黑帧后 AE 增益显著上升，过曝帧 OCR 进一步下降
- 单测：黑帧像素全 0；AE 增益随黑帧占比单调上升

**优先级/工作量**：P0 / 中

---

### A3. 多显示器同步

**现状/问题**
交底书 4.4 节描述主从时钟同步（Master 广播 VBlank，±0.1ms 容差）与异构刷新率处理（240Hz+144Hz→720Hz 虚拟时钟）。当前完全未实现，`timing_controller.py` 只管单显示器。

**改动位置**
- 新建 `src/core/multi_display.py`

**具体方案**
1. `MultiDisplaySync`：一个 Master + 多个 Slave，Master VBlank 时间广播
2. 异构刷新率：求各刷新率 LCM 作虚拟时钟基准，各屏按 `虚拟时隙数/n` 节拍取子帧
3. 同步误差统计：各从屏切换时刻与主屏的偏差

**验证方法**
- 两台模拟屏（240/144Hz）同步误差 <±0.1ms
- 任意相机曝光窗口内两屏均呈碎片态（无完整帧泄露）

**优先级/工作量**：P0 / 中

### A4. 性能实测

**现状/问题**
交底书 6.4 给出一张性能表（GPU 额外负载 3-5%、帧生成 +0.3ms、端到端 +0.5ms、CPU<1%），但这些数字在代码里**从未被测量**，纯属声称。答辩时若被问"这些数据怎么来的"会很被动。

**改动位置**
- 新建 `experiments/performance_benchmark.py`

**具体方案**
测量并如实报告：
1. 掩模生成耗时（`MaskGenerator.generate`，含拒绝采样后）
2. GPU 单子帧渲染耗时（`GPURenderer.render_subframe`）vs 软件渲染
3. 噪声生成耗时
4. 端到端单帧合成延迟、可达 FPS
5. `TimingController` 实测抖动（诚实说明 Python+非实时 OS 的局限）

**验证方法**
- 输出实测性能表，与交底书声称值并列对比，**标注差距与原因**（Python/CPU 渲染 vs 交底书假设的驱动层 GPU）

**优先级/工作量**：P0 / 低（最高性价比）

## 2. P1 — 安全可信度

### B1. 真实对抗噪声 + 消融实验

**现状/问题**（严重）
`noise_injector.py:96-128` 的 `generate_pytorch_fgsm` 用 `models.vgg11(weights=None)` —— **随机初始化、未训练**的网络，其梯度是无意义噪声，不构成任何"对抗攻击"。主路径 `_gradient_based_noise`（行 65-94）只是 Sobel 边缘 + 随机扰动的启发式。因此交底书"创新点 B：基于梯度攻击的对抗噪声"在实现层面**未被验证**。攻击分析已暗示真正起作用的是掩模分割，不是噪声。

**改动位置**
- `src/core/noise_injector.py`：`generate_pytorch_fgsm` 改用预训练权重
- 新建 `experiments/ablation_noise.py`：噪声消融

**具体方案**
1. 预训练模型路径：`weights=VGG11_Weights.DEFAULT`，损失改为对真实视觉特征的有界扰动（或攻击一个预训练 CRNN 文本识别代理）
2. 消融实验：四组对照
   - 仅掩模（无噪声）
   - 仅噪声（无掩模分割）
   - 掩模+启发式噪声（当前）
   - 掩模+真实对抗噪声
   测各组单帧 OCR，量化噪声边际贡献

**验证方法**
- 消融表显示：去掉噪声后单帧 OCR 变化 < X%（预期很小，证明掩模是主力）—— 这是诚实且有价值的结论

**优先级/工作量**：P1 / 中

---

### B2. 去混淆/重构网络攻击实验

**现状/问题**
交底书 7.3 把"AI 自适应去混淆攻击"列为重点防御对象，但实验里**完全没有这种攻击**。`camera_simulator.py` 只有时域平均（已证明能破解）。缺少对"攻击者训练重构网络"的量化。

**改动位置**
- 新建 `src/attack/reconstruction_attack.py`

**具体方案**
实现三档重构攻击，由弱到强：
1. 多帧中值堆叠 + 去噪（无需训练）
2. 单帧 inpainting 重构基线（OpenCV inpaint / 简单插值）
3. （可选）轻量 U-Net 学习"单子帧→原图"映射

**验证方法**
- 量化各档重构后的 OCR 恢复率，诚实呈现哪些攻击有效、哪些无效

**优先级/工作量**：P1 / 中

---

### B3. 视角差异化掩模（抗时域平均缓解）

**现状/问题**
这是 PoC 暴露的根本局限（多帧叠加可还原）的**唯一可行缓解方向**（交底书 7.2）。当前 `mask_generator.py` 全屏用同一掩模序列，任何对齐的相机都能完整采样。

**改动位置**
- `src/core/mask_generator.py`：增视角相关掩模

**具体方案**
1. 空间分区，不同区域用不同子密钥派生掩模序列
2. 建模离轴相机：因 LCD 视角依赖 + 区域掩模差异，离轴相机看到的子帧序列与正视不一致 → 帧对齐失败 → 时域平均无法重构
3. 对比正视 vs 离轴相机的时域平均还原质量

**验证方法**
- 模拟两个视角，离轴视角时域平均后 OCR 显著低于正视

**优先级/工作量**：P1 / 中

## 3. P2 — 实验严谨性

### C1. 掩模取模偏置修复

**现状/问题**
`mask_generator.py:54-56`：`raw[:num_pixels] % self.n`，`raw` 是 uint8（0-255）。当 n 不是 2 的幂（n∈{3,5,6,7}）时存在**系统性取模偏置**——例如 256 mod 3 = 1，值 0 比值 1、2 多出现一次，偏置约 1/256。卡方检验（行 59-64）只能在偏置极端时报警，且重试（行 83 偏移 cycle）改变不了分布形状。这违背交底书"密码学安全均匀掩模"的声称。

**改动位置**
- `src/core/mask_generator.py`：`_generate_index_matrix` 改拒绝采样

**具体方案**
读 ⌈log₂n⌉ 位组成整数，落在 `[n, 2^⌈log₂n⌉)` 则拒绝重采。消除偏置（n=2,4,8,16 无影响，n=3,5,6,7 受益）。

**验证方法**
- 多次生成，各 n 的卡方 p 值应近似 Uniform(0,1)（无系统偏置）

**优先级/工作量**：P2 / 低

### C3. SSIM 感知指标 + 运动清晰度

**现状/问题**

**改动位置**

**验证方法**
- SSIM > 0.98（视觉无损）；运动模糊宽度增幅 <5%

**优先级/工作量**：P2 / 低

## 5. 风险与取舍

| 改进 | 取舍/风险 |
|------|-----------|
| A1 HDR | 真实效果需 HDR 屏（峰值 1000+ nits）；PoC 仅在数值/SDR 屏上模拟验证 |
| A3 多屏 | PoC 是软件时序模拟，±0.1ms 是理论值；真实需硬件 VBlank 同步 |
| A4 性能 | 实测值会**低于**交底书声称（Python+CPU 渲染 vs 驱动层 GPU），需如实标注差距 |
| B1 对抗噪声 | 消融很可能证明噪声贡献小——这对专利是把双刃剑，但科学诚实更重要 |
| B3 视角掩模 | 分区掩模可能略降正视视觉质量，需测 ΔE 代价 |
| C2 多引擎 | EasyOCR 首次运行下载模型、较慢；PaddleOCR 依赖重，PoC 可选做 |

## 6. 实测结果（实现后回填）

### A4 性能实测（`experiments/performance_benchmark.py`）

| 环节 | 1080p | 1440p | 交底书声称 |
|------|-------|-------|-----------|
| 掩模生成（拒绝采样） | 15.3 ms | 27.4 ms | <0.1ms (Compute Shader) |
| 对抗噪声生成 | 103.9 ms | 197.0 ms | — |
| 子帧合成 SDR (×4) | 79.9 ms | 141.9 ms | — |
| 子帧合成 HDR (×4) | 786.9 ms | 1367.2 ms | — |
| GPU 渲染单子帧 | 31.6 ms | 55.0 ms | +0.3ms (n=4) |
| **端到端单周期** | **199.1 ms** | **366.3 ms** | — |
| 240Hz 周期预算 | 16.67ms ✗ 超预算 | 16.67ms ✗ | — |

> **诚实结论**：PoC 的 Python/CPU 路径比交底书声称的驱动层 GPU 实现慢 1-2 个数量级。
> 交底书的 <0.3ms 指标假设 Compute Shader 并行，需将算子下沉至 GPU 驱动/合成器层
> （第四章），属 PoC 范围外。PoC 价值在**算法正确性与防御有效性**，非生产级性能。

### B2 去混淆/重构攻击（`src/attack/reconstruction_attack.py`，n=4 含对抗噪声）

| 重构方法 | PSNR | SSIM | OCR 恢复 | 结论 |
|----------|------|------|----------|------|
| **逐像素 max 堆叠** | **30.4 dB** | **0.991** | **100%** | ✗ **完全攻破** |
| 时域平均+锐化 | 3.7 dB | 0.478 | 100% | ✗ 攻破 |
| 中值堆叠 | 1.1 dB | 0.051 | 38% | 部分 |
| 单帧 inpainting | 2.3 dB | 0.026 | 0% | ✓ 防御有效 |

> **关键发现（比原 PoC 结论更尖锐）**：由于每像素在一个周期内恰好被点亮一次，
> 攻击者对覆盖完整周期的多帧做**逐像素取最大值**即可近乎完美还原（SSIM 0.991，
> OCR 100%），且对抗噪声因被 max 选择性忽略而**完全失效**。这把"多帧时域平均"
> 这一局限进一步坐实为"任何覆盖完整周期的逐像素聚合（max/median/mean）都能反演
> 互斥完备掩模"。单帧 inpainting 因缺失 3/4 像素而失败——单帧防御依然成立。

### B1 对抗噪声消融（`experiments/ablation_noise.py`，n=4）

| 实验组 | 单帧 OCR | 说明 |
|--------|----------|------|
| 基线（原始帧） | 89.2% | — |
| 组1 仅掩模（无噪声） | **0.0%** | 掩模单独即完全防御 |
| 组2 仅噪声（无掩模分割） | **91.9%** | 噪声单独**零防御**（甚至略高于基线） |
| 组3 掩模+启发式噪声(Sobel) | 0.0% | — |
| 组4 掩模+真实对抗噪声(预训练VGG) | 0.0% | — |
| **噪声边际贡献（组1−组3）** | **+0.0%** | — |

> **决定性结论**：随机点阵掩模分割贡献了**全部**防御效果；对抗性噪声（无论
> 启发式还是真实预训练 VGG FGSM）边际贡献为 **0**。这意味着交底书"创新点 B
> 对抗性噪声"在防御层面**不是有效来源**——掩模才是。对论文，这是诚实且有力的
> 消融证据；对专利，建议弱化对"对抗噪声"独立有效性的声称（其价值在视觉无损约束
> ΣN_k=0，而非提升单帧防御）。

### A3 多显示器同步（`src/core/multi_display.py`）
- 同刷新率双屏（240Hz）：VBlank 广播同步误差 max 0.05ms < 0.1ms ✓
- 异构（240+144Hz）：虚拟时钟 LCM=720Hz，靠无完整帧泄露保证

### C2 多样本多引擎评测（`experiments/build_corpus.py` + `benchmark.run_corpus_multi_engine`）

12 张多样化语料（中/英/代码/表格/数字/账号/URL），n=4：

| OCR 引擎 | 原始帧准确率 | 单子帧准确率 |
|----------|-------------|-------------|
| Tesseract | 69.5% ± 40.2% | **0.0% ± 0.2%** |
